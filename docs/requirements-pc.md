# LanCtrl PC 端需求文档

## 1. 目标定位

PC 端是运行在 Windows 上的原生后台控制服务，负责在局域网内向移动端暴露受控能力，并以低资源占用、稳定运行、可持续扩展为第一目标。

PC 端不承担 UI 主职责。即便未来增加托盘、设置窗口，也应与核心服务解耦。

## 2. 核心职责

- 在局域网内被发现与识别
- 提供安全配对和鉴权
- 接收并执行系统控制命令
- 持久化并执行定时任务
- 维护共享目录配置与访问元数据
- 记录日志并推送状态变化

## 3. 技术栈建议

## 3.1 主语言

- **Rust stable**

原因：

- 原生性能高，空闲资源占用低。
- 无 GC，更适合长期常驻后台。
- 对网络、并发、SQLite、Windows API 封装成熟。
- 相比 C++，长期维护成本和内存安全更优。

## 3.2 推荐库组合

- 异步运行时：`tokio`
- HTTP 服务：`axum`
- WebSocket：`axum` / `tokio-tungstenite`
- 序列化：`serde`、`serde_json`
- 配置管理：`config` 或 `figment`
- 本地数据库：`sqlx + SQLite`
- 时间处理：`time`
- 唯一 ID：`uuid`
- 日志：`tracing`、`tracing-subscriber`
- 错误处理：`thiserror`、`anyhow`
- Windows API：`windows`
- 本地服务安装/启动：自研薄封装 + Windows SCM 接口
- 加密：`ring` 或 `rustls` 生态
- mDNS：`mdns-sd`
- UDP 广播补充：`tokio::net::UdpSocket`

不建议首期使用：

- `tauri` 作为核心服务承载层
- `actix-web`
- 自定义线程模型
- 重量级 ORM

原因是首期重点不是桌面 UI，也不是极致开发效率，而是长期稳定和结构清晰。

## 4. 模块划分

建议采用 workspace 或清晰模块结构：

- `lanctrl-service`
- `lanctrl-transport`
- `lanctrl-auth`
- `lanctrl-control`
- `lanctrl-scheduler`
- `lanctrl-share`
- `lanctrl-storage`
- `lanctrl-discovery`
- `lanctrl-common`

职责边界要求：

- `transport` 不直接写业务规则，只做协议适配。
- `control` 不关心 HTTP/WebSocket 来源，只暴露命令接口。
- `scheduler` 只负责任务编排与触发，不直接做网络通信。
- `storage` 负责仓储实现，不反向依赖上层模块。

## 5. 功能需求

### 5.1 服务运行与生命周期

- 支持安装为 Windows 后台服务。
- 支持开机自启动。
- 支持异常重启恢复。
- 支持优雅退出。
- 支持热更新配置文件，首期至少支持重载部分非核心配置。

### 5.2 设备发现

- 周期性广播自身存在或响应发现请求。
- 暴露设备名称、设备 ID、软件版本、支持能力。
- 设备 ID 必须稳定，不依赖当前 IP。
- 支持 mDNS 优先，UDP 广播为兜底。

### 5.3 配对与鉴权

- 首次连接必须经过移动端请求、PC端点击“允许”的弹窗确认流程。
- 配对成功后为移动端生成长久不失效的设备凭证（Token Hash 持久化保管）。
- 必须提供“撤销设备/解除配对”功能，允许从已授权列表中剔除手机。
- 请求鉴权采用 Bearer Token。
- 预留多移动端配对支持。

### 5.4 系统控制命令

首期必须支持：

- 关机
- 重启
- 睡眠
- 查询是否存在待执行关机任务
- 取消待执行关机

控制接口要求：

- 每个命令需要统一的执行结果模型。
- 执行失败时返回可定位原因。
- 系统调用需封装为命令执行器接口，便于扩展更多操作。

### 5.5 定时任务

- 支持相对时间任务。
- 支持绝对时间戳任务。
- 任务需要持久化到本地数据库。
- 服务重启后需恢复未完成任务。
- 任务状态至少包含：待执行、执行中、成功、失败、已取消、已过期。

### 5.6 共享目录管理

- 支持维护允许暴露的共享目录配置。
- 支持返回目录别名、UNC 路径、权限说明、是否可写。
- 不直接承载 SMB 协议实现。
- 如未来需要内嵌浏览能力，应通过新增模块扩展，不影响控制服务主流程。

### 5.7 日志与审计

- 记录命令执行日志。
- 记录任务创建、取消、执行结果。
- 记录配对与鉴权事件。
- 记录设备连接与异常信息。
- 日志分为结构化运行日志与业务审计日志两类。

## 6. API 设计建议

首期建议使用 REST + WebSocket。

### 6.1 REST

用途：

- 设备配对
- 控制命令下发
- 任务创建/取消/查询
- 共享目录元数据获取
- 日志查询

### 6.2 WebSocket

用途：

- 实时在线状态
- 任务状态更新
- 控制命令执行结果推送
- 心跳与连接保活

### 6.3 版本策略

- 接口统一加 `/api/v1`
- 返回结构统一
- 错误码体系独立定义
- 破坏性变更必须走新版本

## 7. 数据存储建议

本地持久化首选 SQLite。

建议表：

- `paired_clients`
- `devices_meta`
- `scheduled_tasks`
- `operation_logs`
- `share_entries`
- `service_settings`

设计要求：

- 任务表需要索引执行时间和状态。
- 日志表需要索引时间和操作类型。
- 凭证不能明文存储。

## 8. 安全要求

- 默认只监听局域网或配置允许的网卡。
- 必须限制未配对访问。
- 配对码有效期短。
- 凭证必须支持本地吊销。
- 所有高风险命令应具备审计记录。
- 后续可扩展“仅允许白名单子网访问”。

## 9. 性能与稳定性要求

- 空闲 CPU 占用接近 0。
- 内存常驻控制在轻量级水平。
- 不允许通过频繁轮询造成持续高 CPU。
- 调度器应采用事件驱动或精确唤醒策略。
- 网络服务异常不能拖垮任务调度模块。

## 10. 大致实现方式

### 10.1 进程模型

建议采用“单后台服务 + 可选前端设置/托盘入口”的模型：

- 常驻核心服务：负责所有网络、任务、控制能力。必须针对开机常驻优化性能，保持绝对轻量的空闲资源占用。
- 可选轻量前端：负责接受配对请求时的“允许弹窗”，以及提供一个管理入口供用户主动【移除已信任设备】。

这样可以保证：

- 服务与 UI 解耦
- 后期改托盘、改设置页不影响核心逻辑
- 部署模型简单

### 10.2 控制命令实现

- 使用 `windows` crate 调用原生 API，必要时配合系统命令。
- 所有系统操作统一收口到 `CommandExecutor` 抽象。
- 每类操作实现成独立 handler，例如 `ShutdownExecutor`、`RestartExecutor`。

### 10.3 定时任务实现

- 移动端只负责创建任务。
- PC 端在本地保存任务并负责最终触发。
- 调度器以最近执行时间构建最小唤醒机制，而不是秒级全量轮询。
- 执行前再次确认任务状态和设备权限。

### 10.4 共享目录实现

- 服务端仅维护可暴露共享目录配置和说明。
- 返回 UNC 路径给移动端。
- 若移动端选择系统方式打开，则交由系统或插件处理 SMB。
- 这样可以避免 PC 端服务承担文件传输与大流量会话压力。

## 11. 可扩展性设计要求

- 命令执行采用注册式能力表，新增命令时不修改主分发核心。
- 调度器支持未来接入重复任务、脚本任务。
- 鉴权模块支持未来增加权限级别。
- 传输层支持未来新增 gRPC，不破坏业务服务接口。

## 12. 不建议的方案

- 不建议用 Electron 或 Flutter Desktop 承载核心后台服务，资源占用与稳定性目标不匹配。
- 不建议把所有逻辑都写进单一 HTTP 服务文件，后续扩展成本过高。
- 不建议把定时关机直接依赖 Windows `shutdown -t` 作为唯一任务机制，应有本地任务模型，否则取消、恢复、查询都不可靠。
