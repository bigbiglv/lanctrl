# LanCtrl 总需求文档

## 1. 文档目标

本文档用于定义 LanCtrl 项目的整体产品范围、系统边界、核心业务流程、通用非功能要求，以及 PC 端与移动端的职责划分，作为后续详细设计、开发排期和接口定义的基础。

LanCtrl 的核心定位不是“远程桌面”产品，而是“同一局域网内对多台 Windows 电脑进行轻量化控制与资源访问的控制系统”。

## 2. 产品目标

### 2.1 目标

- 支持移动端在同一局域网内发现并管理多台 Windows 电脑。
- 支持对电脑执行轻量控制操作，例如关机、重启、睡眠、唤醒能力预留。
- 支持访问电脑预先开放的共享文件夹。
- 支持立即执行和定时执行控制任务。
- 保证电脑端后台资源占用低、稳定运行、易扩展。
- 保证移动端操作直观、视觉美观、后续 UI 改版时不需要重构业务层。

### 2.2 非目标

- 不实现类似向日葵、ToDesk、TeamViewer 的屏幕级远程控制。
- 不实现跨公网直连或中继服务。
- 不自研文件传输协议，优先依赖 Windows 原生共享能力。
- 不以“快速上线”为第一优先级，优先保证架构扩展性和长期维护性。

## 3. 系统范围

系统由两个核心子系统组成：

- PC 端：运行在 Windows 上的轻量级后台原生服务，负责设备发现、鉴权、命令执行、任务调度、共享目录元数据提供。
- 移动端：基于 Flutter 的控制应用，负责设备管理、指令下发、任务配置、共享资源入口展示、日志与状态展示。

## 4. 典型使用场景

### 4.1 设备控制

- 用户在手机上看到已配对的电脑列表。
- 用户选择某台电脑并执行关机、重启、睡眠等操作。
- 用户可查看执行状态、错误原因和最近操作记录。

### 4.2 多设备管理

- 用户可以添加多台电脑。
- 每台电脑有独立名称、备注、在线状态、最近连接时间、能力集。
- 用户可以对设备进行分组、排序、重命名，该能力作为后续扩展预留。

### 4.3 定时控制

- 用户可设置“2 小时后关机”等相对时间任务。
- 用户可设置“发送时间戳给电脑，在指定绝对时间关机”的任务。
- 任务由电脑端本地持久化并执行，避免移动端退出后任务失效。

### 4.4 共享资源访问

- 用户在移动端查看目标电脑暴露的共享文件夹信息。
- 用户通过系统支持的 SMB 能力访问共享目录。
- 项目本身负责提供目录入口与权限说明，不自研文件内容同步协议。

## 5. 核心功能需求

### 5.1 局域网设备发现

- 支持同一局域网内设备发现。
- 支持手动输入 IP 直连作为发现失败时的补充方式。
- 支持设备在线/离线状态识别。
- 支持设备唯一标识，避免 IP 变化后丢失设备关系。

### 5.2 安全配对与鉴权

- 首次连接必须显式配对。
- 采用移动端请求配对、PC端点击确认的授权机制。
- 配对成功后生成长期有效、永不失效的设备凭证（Token）。
- 建立双端设备管理/解绑机制，支持从移动端移除 PC 或从 PC 端移除移动端。
- 后续所有控制请求必须基于凭证鉴权。
- 考虑到家用场景环境内 IP 是动态的，凭证管理依托设备唯一标识(deviceId)，每次连接前自动以最新扫描的 IP 为准。

### 5.3 设备控制

首期建议纳入以下控制项：

- 立即关机
- 立即重启
- 立即睡眠
- 取消待执行关机

二期预留：

- 锁屏
- 注销当前用户
- 唤醒能力联动（依赖局域网唤醒方案）
- 自定义脚本任务

### 5.4 定时任务

- 支持相对时间任务：例如 N 分钟后关机。
- 支持绝对时间任务：例如某个时间戳执行关机。
- 支持查看待执行任务。
- 支持取消任务。
- 支持任务执行结果记录。

### 5.5 共享文件夹访问

- 支持展示电脑开放的共享目录清单。
- 支持展示访问路径、说明、权限信息。
- 支持移动端直接调用系统/插件能力访问 SMB 共享。
- 不将共享浏览与控制命令耦合在同一业务模块内。

### 5.6 日志与状态

- 记录每次控制请求的发起设备、目标设备、操作类型、发起时间、执行结果。
- 展示设备在线状态、最近心跳时间、最近错误。
- 日志能力必须可扩展为后续审计模块。

## 6. 核心业务流程

### 6.1 首次配对流程

1. 移动端发现局域网中的 PC 服务。
2. 用户选择候选设备发起配对请求，手机进入等待确认状态。
3. PC 端弹出授权提示，用户点击“允许”。
4. 双方完成配对并生成长久不过期的设备凭证。
5. 移动端将设备保存为已管理设备。

### 6.2 立即控制流程

1. 用户选择设备。
2. 用户触发控制操作。
3. 移动端发送已签名请求。
4. PC 端鉴权后调用系统能力执行。
5. PC 端返回结果并写入日志。
6. 移动端刷新状态与最近记录。

### 6.3 定时任务流程

1. 用户选择目标设备和任务类型。
2. 用户选择相对时间或绝对时间。
3. 移动端下发任务。
4. PC 端落库并由本地调度器执行。
5. 到时执行后更新任务状态并写入日志。

## 7. 通用非功能要求

### 7.1 性能

- PC 端常驻内存占用尽量低，空闲 CPU 占用接近 0。
- 移动端常用页面必须在低网络波动场景下保持可操作。
- 设备列表、日志列表、任务列表需要支持分页或增量加载。

### 7.2 稳定性

- PC 端服务需支持开机自启。
- PC 端异常退出后可自动恢复。
- 定时任务需具备断电/重启后恢复能力。

### 7.3 安全性

- 所有控制接口必须鉴权。
- 凭证必须安全存储。
- 默认仅允许局域网访问。
- 敏感操作建议支持二次确认和操作审计。

### 7.4 可扩展性

- 设备发现、鉴权、控制命令、调度任务、共享目录应采用模块化边界。
- API 设计需要具备版本化能力。
- 移动端 UI 层与业务层必须解耦，后续换皮或改交互不影响核心逻辑。

## 8. 推荐通信方案

首期建议采用以下组合，而不是一开始引入过重协议：

- 设备发现：mDNS 或 UDP 广播
- 控制 API：HTTP REST
- 实时状态推送：WebSocket
- 文件访问：SMB（系统能力）

这样做的原因：

- 比纯 WebSocket 全栈更容易调试与版本演进。
- 比 gRPC 在 Flutter 与 Windows 原生首期集成成本更低。
- REST + WebSocket 的组合足够覆盖当前场景，且后续平滑演进空间大。

## 9. 总体技术选型原则

### 9.1 PC 端

- 必须使用原生开发。
- 以性能、稳定性、低资源占用为第一优先级。
- 以模块边界清晰、便于新增控制能力为第二优先级。
- 推荐使用 Rust 作为主语言，而不是 C# / Electron / Java。

原因：

- Rust 能兼顾原生性能、内存安全、生态成熟度和后续长期维护成本。
- 对后台服务、网络通信、任务调度、SQLite 持久化都已有成熟方案。

### 9.2 移动端

- 使用 Flutter。
- 以交互流畅、视觉质量、组件化和未来 UI 改版可维护性为第一优先级。
- 状态管理、路由、依赖注入、网络层、数据模型必须采用社区主流可扩展方案，不做临时拼装。

## 10. 建议的系统分层

### 10.1 PC 端

- `service-core`：服务启动、生命周期管理
- `discovery`：局域网发现与广播
- `auth`：配对、令牌、权限
- `control`：关机/重启/睡眠等命令执行
- `scheduler`：定时任务调度与恢复
- `share`：共享目录配置与元数据
- `storage`：SQLite 持久化
- `transport`：HTTP/WebSocket 接口层
- `logging`：操作日志、运行日志

### 10.2 移动端

- `presentation`：页面、组件、交互状态
- `application`：用例编排、任务入口
- `domain`：实体、仓储接口、业务规则
- `data`：网络请求、本地缓存、DTO 映射
- `core`：主题、导航、错误处理、基础组件

## 11. 里程碑建议

### M1 最小可用版本

- 单局域网设备发现
- 单设备配对
- 多设备管理
- 立即关机 / 重启 / 睡眠
- 相对时间关机
- 基础日志

### M2 可用增强版本

- 绝对时间任务
- 任务取消与恢复
- 共享目录列表
- WebSocket 状态推送
- 更完整的异常提示与审计信息

### M3 扩展版本

- 设备分组
- 权限分级
- 更多系统控制命令
- Wake-on-LAN 联动
- 插件式自定义任务

## 12. 风险与设计提醒

- **网络环境边界限制**：设备发现高度依赖 mDNS。需注意：
  - Windows 防火墙：极易拦截 3000 端口和 UDP 广播，首次安装需引导用户放行。
  - iOS 本地网络权限：iOS 14+ 强制要求声明 `NSLocalNetworkUsageDescription` 等，需妥善处理拒绝对话框。
  - AP 隔离：部分企业或公共路由器开启了 AP 隔离，会导致搜寻不到设备。需要在 UI 提供排错说明。
- **动态局域网 IP 处理**：虽然标识设备靠持久化 `deviceId`，但底层通讯仍靠动态 `IP:Port`。局域网 IP 会变，移动端必须在每次通过扫描拿到已知 `deviceId` 后，立即静默更新该设备的最新 IP。
- “访问共享文件夹”如果要求在 Flutter 内做完整文件浏览器，落地复杂度会明显提升，需要尽早确认是否接受“跳系统文件管理器/使用 SMB 插件”的方案。
- Windows 后台服务如果要执行关机、睡眠等操作，必须提权或验证运行策略。考虑到常驻后台定位，必须严格优化控制空闲时 CPU 和内存占用。
- 若未来要支持跨公网，当前鉴权、发现、连接模型都要扩展，但不建议为该需求提前复杂化当前架构。
