# LanCtrl 移动端需求文档

## 1. 目标定位

移动端是用户与局域网内多台电脑交互的主要入口，核心目标是：

- 操作足够简单
- 信息结构清晰
- UI 美观且具备产品感
- 后续 UI 改版时不需要重构业务逻辑

因此移动端必须采用严格分层、组件化主题体系和稳定的状态管理方案，而不是页面直连接口的快速开发方式。

## 2. 核心职责

- 发现并添加局域网内电脑
- 管理多台电脑
- 发起控制命令
- 创建与管理定时任务
- 展示共享目录入口
- 展示设备状态、任务状态和日志

## 3. 技术栈建议

### 3.1 基础框架

- **Flutter（稳定版）**
- **Dart**

### 3.2 推荐库组合

- 状态管理：`Riverpod`（推荐使用 `flutter_riverpod` + `riverpod_generator`）
- 路由：`go_router`
- 数据模型：`freezed` + `json_serializable`
- 网络层：`dio`
- WebSocket：`web_socket_channel` 或基于 `dio` 独立封装
- 本地存储：`drift` 或 `isar`
- 安全存储：`flutter_secure_storage`
- 国际化：`slang` 或 Flutter 官方 `gen_l10n`
- 日志与调试：`logger`
- 设计系统辅助：`theme_tailor` 或基于 `ThemeExtension` 自建方案
- 表单：`reactive_forms` 或轻量自建表单状态封装
- 列表/刷新：社区成熟刷新组件，但避免侵入业务层

不建议首期使用：

- `GetX` 作为全家桶
- 业务代码大量依赖 `BuildContext`
- 页面层直接持有 `dio` 请求逻辑

## 4. 架构要求

建议采用分层清晰的 feature-first + clean architecture 混合结构：

- `core`
- `shared`
- `features/device`
- `features/control`
- `features/schedule`
- `features/share`
- `features/log`
- `features/settings`

每个 feature 内部分层：

- `presentation`
- `application`
- `domain`
- `data`

这样做的价值：

- UI 换版只改 `presentation`
- 接口变化主要落在 `data`
- 业务规则集中在 `domain`
- 页面复杂度增长后不容易失控

## 5. 功能需求

### 5.1 设备列表

- 展示所有已添加电脑
- 展示在线/离线状态
- 展示设备名、备注、最近活动时间
- 支持手动刷新和自动刷新
- 支持设备置顶、排序、重命名，首期可只做结构预留

### 5.2 添加设备

- 支持自动发现局域网设备
- 支持手动输入 IP 添加
- 支持输入配对码完成绑定
- 配对成功后保存设备凭证

### 5.3 设备详情

- 展示设备基础信息
- 展示支持能力
- 展示最近任务和最近操作
- 展示共享目录入口

### 5.4 控制操作

首期必须支持：

- 关机
- 重启
- 睡眠
- 取消待执行关机

交互要求：

- 危险操作二次确认
- 执行中状态清晰可见
- 成功/失败反馈明确

### 5.5 定时任务

- 支持“X 分钟/小时后执行”
- 支持“选择具体时间执行”
- 支持查看任务列表
- 支持取消任务
- 支持展示任务执行结果

### 5.6 共享目录访问

- 展示共享目录卡片列表
- 显示名称、说明、权限、路径
- 支持一键复制路径
- 支持调用系统能力或插件打开 SMB 路径

### 5.7 日志与状态

- 展示最近控制日志
- 展示最近任务执行记录
- 展示设备连接状态变化

## 6. UI/UX 设计要求

### 6.1 总体原则

- 以“控制中心”而不是“工具列表”的感觉来设计。
- 首页必须优先展示设备状态和高频操作。
- 危险操作要有明显层级，不和普通操作混排。
- 任务与即时控制要区分，但流程不能割裂。

### 6.2 推荐信息架构

建议底部主导航：

- 设备
- 任务
- 日志
- 设置

设备详情页建议分区：

- 设备概览
- 快捷控制
- 定时任务
- 共享目录
- 最近记录

### 6.3 视觉系统要求

- 主题色、语义色、间距、圆角、阴影、字体层级必须统一管理。
- 页面只依赖设计系统 token，不直接写散落颜色和尺寸。
- 组件要有稳定状态模型：默认、悬浮、按下、禁用、危险、成功、警告。

### 6.4 为未来改版预留

必须避免以下问题：

- 页面直接依赖具体颜色和尺寸
- 业务逻辑嵌入 widget 树
- 同一类卡片组件写多个相似版本

建议做法：

- 所有设计 token 收敛到 `ThemeExtension`
- 常用容器、按钮、状态标签、列表卡片做成基础组件
- 页面通过 view model 驱动，不直接拼数据流程

这样后续即便改成拟物、极简、深色、企业风，只需要改主题层和组件实现，不需要重写业务。

## 7. 数据流与状态管理要求

推荐采用：

- `Notifier/AsyncNotifier` 管理页面状态
- Repository 抽象设备、任务、日志、共享目录访问
- DTO 与 Domain Entity 分离
- 页面只关心 UI State，不直接依赖原始接口结构

典型数据流：

1. 页面触发 action
2. Provider 调用 use case
3. Use case 调用 repository
4. Repository 选择远端接口或本地缓存
5. 结果映射为 UI state

## 8. 本地存储要求

需要持久化的数据：

- 已配对设备列表
- 设备凭证
- 最近连接记录
- 用户偏好设置
- 可离线查看的部分日志/任务缓存

安全数据必须进入安全存储：

- Token
- 配对凭证

普通结构化缓存可进入本地数据库。

## 9. 网络通信要求

- REST 请求统一通过 `dio` 封装
- Token 刷新或失效处理统一拦截
- WebSocket 连接状态要能同步到全局
- 离线、超时、鉴权失败、服务不可达要区分提示

## 10. 大致实现方式

### 10.1 首页/设备列表

- 通过局域网扫描与已保存设备合并展示。
- 在线设备实时刷新，离线设备保留历史信息。
- 卡片上展示快捷操作入口，但危险操作仍需确认。

### 10.2 添加设备流程

- 发现页展示候选设备。
- 用户选择设备后进入配对页。
- 配对成功写入本地数据库和安全存储。

### 10.3 控制操作流程

- 设备详情页调用控制用例。
- 成功后刷新设备状态和最近日志。
- 失败时展示明确错误，如鉴权失败、设备离线、命令拒绝。

### 10.4 定时任务流程

- 移动端只负责创建、查询、取消。
- 真正调度和执行由 PC 端负责。
- 移动端展示任务状态机，不承担本地定时器逻辑。

### 10.5 共享目录访问流程

- 先读取 PC 返回的共享目录元数据。
- 再通过系统能力或插件打开 SMB 路径。
- 如果平台限制导致不能直接打开，需优雅降级为复制路径和提示说明。

## 11. 可扩展性要求

- 多设备、多任务、多日志列表都必须支持后续分页和筛选。
- 设备能力要做成能力枚举或能力映射，不把支持项写死在 UI。
- 控制项 UI 根据设备能力动态渲染，方便后续新增命令。
- 页面结构应支持未来增加平板布局。

## 12. 不建议的方案

- 不建议把页面、状态管理、接口请求写在同一个文件。
- 不建议把设备详情页做成超大 `StatefulWidget`。
- 不建议先做一套临时 UI 再重构，因为这类项目一旦把业务和界面耦合，后面改版成本会非常高。
- 不建议过度依赖第三方 UI 模板库，应建立自己的设计系统层。
